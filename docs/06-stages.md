# 06. 단계별 목표 상세

본 문서는 1~4차 목표 각각에 대해 다음 정보를 제공한다:

- 산출물 정의
- 동작 시나리오 (사용자 관점)
- 구현 컴포넌트 및 책임
- 완료 기준 (Definition of Done)
- 위험 요소

## 1차 — 컨베이어 비전 검출 + 서보 분류

### 산출물

- 컨베이어 벨트가 동작하며 RGBY 색상 물체가 흐른다
- RealSense 카메라가 위에서 컨베이어를 촬영
- 빨간색(R) 물체만 불량으로 판정, 나머지(G/B/Y)는 정상
- 컨베이어 끝의 서보 분류기가 갈래를 결정 (정상 → 고정 박스, 불량 → 터틀봇 박스 위치)
- 검출 결과가 SQLite에 저장되고 웹 UI 차트에 실시간 표시
- 영상은 검출 박스 오버레이된 상태로 웹 UI에 스트리밍

### 동작 시나리오

1. 사용자가 웹 UI에서 "컨베이어 시작" 버튼 클릭
2. Server가 컨베이어 라즈베리파이에 HTTP POST `/conveyor/start`
3. 라즈베리파이가 스테퍼 모터에 펄스를 보내 벨트 회전 시작
4. 사용자가 RGBY 물체를 컨베이어 위에 올림
5. RealSense 카메라가 매 프레임 캡처 (15~30 fps)
6. Vision Node가 HSV 변환 후 빨간색 mask로 검출
7. 검출 결과를 ROS2 토픽 `/defect/detection`에 publish
8. Server가 토픽 구독 → DB에 detections row 추가 → WebSocket으로 UI에 push
9. Server가 컨베이어 라즈베리파이에 분류 명령 HTTP POST (`/sort/normal` 또는 `/sort/defect`)
10. 라즈베리파이가 서보를 해당 갈래로 회전
11. 물체가 컨베이어 끝에 도달하여 분류된 갈래로 떨어짐
12. UI 차트에 누적 검사 수, 불량 수, 불량률이 실시간 갱신

### 구현 컴포넌트

| 컴포넌트 | 위치 | 책임 |
|---|---|---|
| Vision Node | 노트북 (ROS2) | RealSense 토픽 구독, HSV 검출, 결과 발행, MJPEG 스트림 제공 |
| Server | 노트북 (ROS2 + FastAPI) | 토픽 구독, DB 저장, 분류 명령 디스패치, WebSocket 푸시 |
| Conveyor HTTP Slave | 라즈베리파이 5 | Flask 서버, gpiozero로 스테퍼/서보 제어, GPIO 입력 처리 |
| Web UI | Browser | WebSocket 구독, 차트, 영상, 제어 버튼 |
| SQLite DB | 노트북 | sessions, detections 테이블 |

### 완료 기준 (Definition of Done)

- [ ] 컨베이어 시작/정지 버튼이 실제 모터를 제어
- [ ] RealSense 영상이 웹 UI에 표시 (검출 박스 오버레이 포함)
- [ ] 빨간색 물체가 컨베이어 위를 지나갈 때 95% 이상의 비율로 검출됨 (정성적 평가)
- [ ] 검출 결과가 0.5초 이내에 분류기 동작으로 이어짐
- [ ] 검출 1건당 SQLite에 1 row 저장 확인
- [ ] UI 차트가 실시간으로 갱신 (누적 카운트, 불량률)
- [ ] 안전 STOP 버튼이 동작 (컨베이어 즉시 정지)

### 위험 요소

- **컨베이어 보드 핀 매핑 미확인**: W1 D1-2에 매뉴얼 수령 후 즉시 검증
- **빨간색 검출이 조명에 따라 불안정**: HSV 임계값 튜닝, 흰색 LED 조명 추가
- **서보 동작 타이밍 불일치**: 컨베이어 속도 측정 후 분류 트리거 시점 보정
- **RealSense USB 대역폭**: 노트북 USB 3.0 포트 필수, 다른 USB 장치와 분리

## 2차 — 터틀봇 연동 운반

### 산출물

- 터틀봇3 와플이 ROS2 네트워크에 합류하여 노트북에서 발견됨
- 터틀봇이 컨베이어 옆 home 위치에서 대기 (불량품 박스를 위에 적재)
- 출동 트리거 두 종류 동작:
  - 음성 명령 ("불량품 비워줘")
  - 누적 카운트 임계 (예: 불량품 5개 쌓이면 자동)
- 터틀봇이 home → 적재장 → home 사이클 자율 이동
- 미션 이력이 robot_missions 테이블에 저장
- UI에 터틀봇 현재 위치 및 미션 상태 표시

### 동작 시나리오

1. 컨베이어 동작 중 빨간 물체가 검출되어 터틀봇 박스에 적재되기 시작
2. 누적 불량품 카운트가 임계 도달 → Server가 자동 출동 트리거
3. 또는 사용자가 "불량품 비워줘" 라고 음성으로 발화
4. Server가 Nav2 액션 클라이언트를 통해 적재장 goal 발행
5. 터틀봇이 Nav2 경로 계획 후 주행 시작
6. UI에 터틀봇 상태가 "moving"으로 표시, 위치가 실시간 갱신
7. 터틀봇이 적재장 도착 → "arrived" 상태
8. (사람이 박스를 비운다고 가정, 또는 단순히 일정 시간 대기)
9. Server가 home 복귀 goal 발행
10. 터틀봇이 home 도착 → "completed" 상태, robot_missions 행 업데이트
11. 컨베이어 적재 카운터 리셋

### 구현 컴포넌트

| 컴포넌트 | 위치 | 책임 |
|---|---|---|
| turtlebot3_bringup | 터틀봇 라즈베리파이 | 라이다, 모터, 오도메트리 publish |
| Nav2 stack | 터틀봇 라즈베리파이 | 경로 계획, 로컬 플래닝, 컨트롤 |
| Server (Mission Manager) | 노트북 | 카운트 모니터링, 트리거, goal 발행, 상태 추적 |
| 음성 인터페이스 | 노트북 + Tailscale → 집 Desktop | STT, LLM 의도 추출 |
| UI 미션 패널 | Browser | 터틀봇 위치 시각화, 미션 상태 표시 |

### 완료 기준

- [ ] 노트북에서 `ros2 topic list`로 터틀봇 토픽 확인 (`/scan`, `/odom`, `/cmd_vel` 등)
- [ ] 노트북에서 발행한 cmd_vel로 터틀봇 수동 조작 가능
- [ ] Nav2가 동작하고 RViz에서 단일 goal 입력 시 자율 주행
- [ ] Server가 미션 트리거 → goal 발행 → 도착 인식까지 자동 흐름 완성
- [ ] 음성 명령으로 동일 흐름 동작
- [ ] 카운트 임계로 동일 흐름 동작
- [ ] robot_missions 테이블에 미션 1회당 1 row 저장

### 위험 요소

- **터틀봇 와플 펌웨어/OpenCR 펌웨어 호환성**: ROS2 Humble용 펌웨어 사전 업데이트 필요
- **Wi-Fi 다중 머신 ROS2 통신 불안정**: ROS_DOMAIN_ID 통일, 같은 서브넷 확인, FastDDS 또는 CycloneDDS 둘 중 하나로 통일
- **카운트 임계 트리거가 음성 명령과 충돌**: 트리거 큐를 두고 한 번에 한 미션만 진행하도록 mutex 설계
- **음성 인식 정확도**: 명령어를 단순한 키워드로 한정 ("비워줘", "복귀", "정지" 등)

## 3차 — SLAM 매핑 + 자율주행

### 산출물

- slam_toolbox로 현장 매핑 완료, `.pgm` + `.yaml` 맵 파일 저장
- 저장된 맵 위에서 Nav2 AMCL 로컬리제이션 동작
- 적재장과 home 위치가 맵 좌표로 등록되어 미션 시 사용
- (시간 여유 시) 동적 장애물 회피 시연 — 사람이 경로에 들어와도 우회

### 동작 시나리오

#### 매핑 단계 (사전 1회 수행)

1. 빈 맵 상태에서 slam_toolbox 실행
2. 사용자가 텔레옵으로 터틀봇을 수동 조작하며 공간 탐색
3. RViz에서 맵이 점진적으로 그려지는 것 확인
4. 충분히 그려지면 `ros2 service call /slam_toolbox/save_map ...`로 저장
5. 적재장과 home 좌표를 RViz에서 클릭하여 측정, 설정 파일에 기록

#### 운영 단계 (매핑 후 매번)

1. 시스템 시작 시 저장된 맵 로드 + AMCL 시작
2. 터틀봇이 초기 위치(home)에서 로컬리제이션 수렴 대기
3. 미션 트리거 시 적재장 좌표를 goal로 발행
4. Nav2가 global plan 생성 → local planner가 실행
5. 동적 장애물 출현 시 local costmap이 즉시 반응 → 우회 경로 생성
6. 우회 불가능 시 global replan
7. 도착 후 home으로 복귀 goal 발행

### 구현 컴포넌트

| 컴포넌트 | 위치 | 책임 |
|---|---|---|
| slam_toolbox | 터틀봇 라즈베리파이 또는 노트북 | 매핑 (사전 단계) |
| Nav2 + AMCL | 터틀봇 라즈베리파이 | 로컬리제이션 + 자율주행 |
| 맵 파일 (.pgm/.yaml) | 공유 (Git에 commit) | 매핑 결과 저장 |
| 좌표 설정 파일 | Server | home, 적재장 좌표 |
| (선택) 동적 장애물 평가 스크립트 | 노트북 | 튜닝 전후 비교 |

### 완료 기준

- [ ] 매핑 완료, 저장된 맵 파일이 Git에 포함
- [ ] AMCL 로컬리제이션 수렴 (RViz에서 위치 점이 안정적으로 한 점에 모임)
- [ ] 시스템 시작 → 미션 트리거 → 적재장 도착 → 복귀의 전체 사이클이 자율로 동작
- [ ] (선택) 동적 장애물 시연 영상 — 사람이 들어왔을 때 우회하는 모습
- [ ] (선택) costmap inflation_radius 등 파라미터 튜닝 전후 경로 비교 자료

### 위험 요소

- **SLAM이 좁은 공간에서 단조로운 맵을 만듦**: 5m 이상 공간 확보 시도, 백업으로 3x3m
- **AMCL 초기 로컬리제이션 실패**: 초기 위치를 RViz에서 수동 입력하는 절차를 시연 시나리오에 포함
- **동적 장애물 회피 실패**: 시간 여유 없으면 단순 자율주행만으로 마감 (체크포인트 W3말 강제)
- **맵 저장 후 가구 위치 변동**: 맵 다시 만들기 (10분 작업)

## 4차 — Dobot Pick & Place (도전 영역)

본 단계는 stretch goal이다. 1~3차 + STT/LLM 통합이 W3말까지 안정적으로 동작할 때만 W4에서 시도한다.

### 산출물

- Dobot 로봇 팔이 컨베이어 옆에 고정 설치되어 ROS2 네트워크에 합류
- 1차에서 사용한 서보 분류기가 Dobot으로 대체됨
- (최소) 정지 상태 물체에 대한 Pick & Place 동작
- (도전) 컨베이어가 동작 중인 상태에서 움직이는 물체 추적 + 픽업

### 동작 시나리오 (정지 상태 픽업)

1. 컨베이어가 검출 영역에서 물체 검출 시 일시 정지
2. Vision Node가 물체의 (x, y) 좌표를 컨베이어 좌표계에서 산출
3. 카메라 ↔ Dobot 좌표 변환 (사전 calibration 결과 적용)
4. Dobot이 해당 좌표로 이동 → 그리퍼 닫기 → 들어올리기
5. 미리 정의된 분류 위치(터틀봇 박스 또는 정상품 박스)로 이동 → 그리퍼 열기
6. home 자세로 복귀
7. 컨베이어 재개

### 동작 시나리오 (움직이는 물체 픽업, 도전)

1. 컨베이어가 동작 중인 상태로 물체 검출
2. 검출 시점의 위치 + 컨베이어 속도로 픽업 시점의 위치 예측
3. Dobot이 예측 위치로 이동, 타이밍에 맞춰 그리퍼 닫기
4. 성공 여부 확인 (그리퍼 압력 센서 또는 비전 재확인)
5. 분류 위치로 이동
6. 실패 시 다음 물체 시도

### 구현 컴포넌트

| 컴포넌트 | 위치 | 책임 |
|---|---|---|
| Dobot Node | 노트북 (ROS2) | Dobot SDK 또는 ROS2 패키지로 제어, 좌표 변환 |
| Calibration 스크립트 | 일회성 | 카메라 ↔ Dobot 좌표 매핑 |
| 추적 알고리즘 | Vision Node 확장 | 컨베이어 속도 추정, 픽업 시점 예측 |

### 완료 기준 (최소 — 정지 픽업)

- [ ] Dobot이 ROS2 명령으로 임의 좌표 이동 가능
- [ ] 카메라 좌표 → Dobot 좌표 변환 검증 (오차 1cm 이내)
- [ ] 정지 물체 1개를 Pick & Place 자동 수행 (성공률 80% 이상)

### 완료 기준 (도전 — 움직이는 픽업)

- [ ] 컨베이어 동작 중 빨간 물체 픽업 (성공률 50% 이상이면 시연 가능 수준)
- [ ] 시연 영상 1회분 이상 확보

### 위험 요소

- **시간 부족으로 W4 안에 시도조차 못 함**: stretch goal이므로 포기 가능, 컨셉만 발표 자료에 포함
- **Calibration 정확도**: 카메라 위치 고정 후 한 번 보정, 시연 직전 재측정
- **움직이는 픽업 알고리즘 복잡도**: 시간 여유 거의 없으면 정지 픽업만으로 마감
- **Dobot SDK가 ROS2 Humble 비호환**: 해당 시 직접 시리얼/USB로 제어하는 wrapper 작성

## 단계 간 의존성

```
1차 (컨베이어 + 검출 + 분류) ─┐
                              │
2차 (터틀봇 연동) ────────────┼──→ 3차 (SLAM + 자율주행) ──→ 4차 (Dobot 도전)
                              │
STT/LLM (1차와 병렬 진행) ────┘
```

- 1차와 2차는 독립적으로 시작 가능 (인터페이스만 합의되면)
- 3차는 2차 완성 후 본격 시작
- 4차는 1~3차 모두 안정화된 후
- STT/LLM은 1차와 병렬로 개발하여 2차에서 통합

## 단계 축소 정책

체크포인트(`07-roles-and-schedule.md` 참조)에서 미달 시 다음 순서로 축소:

1. **W2말 미달**: 4차 즉시 포기 선언
2. **W3말 미달**: 3차의 동적 장애물 회피 포기, 기본 자율주행만 유지
3. **W4 중반 미달**: STT/LLM의 일부 명령 카탈로그 생략, 핵심 명령(정지, 출동)만 유지
4. **최후**: 사전 녹화 영상으로 미완성 부분 보완하지 않고, 솔직한 결과물로 마감
