# 10. 위험 요소 및 대응

## 위험 매트릭스

영향(Impact)과 발생 가능성(Likelihood) 기준으로 위험을 분류하고 대응 방안을 명시한다.

### High Impact / High Likelihood (집중 관리)

| # | 위험 | 영향 | 대응 |
|---|---|---|---|
| R1 | 컨베이어 보드 핀 매핑 미확인 | 1차 시작 자체 불가 | W1 D1에 매뉴얼 수령 또는 멀티미터 추적, 미확정 시 즉시 일정 재조정 |
| R2 | ROS2 다중 머신 통신 불안정 | 2~3차 진행 막힘 | ROS_DOMAIN_ID 통일, 같은 서브넷 5GHz Wi-Fi, FastDDS/CycloneDDS 통일 |
| R3 | 색 검출 조명 민감도 | 1차 검출 정확도 저하 | 백색 LED 추가 조명, HSV 임계값 환경별 튜닝, 시연 환경 고정 |
| R4 | 4주 일정 빠듯함 | 결과물 미완성 | 체크포인트 강제, 4차 즉시 포기 가능 구조, 시연 영상 W3 사전 촬영 |

### High Impact / Low Likelihood (예방 + 백업 계획)

| # | 위험 | 영향 | 대응 |
|---|---|---|---|
| R5 | 시연 당일 하드웨어 고장 | 결과물 시연 불가 | 라이브 시연 안 함, 사전 녹화 영상으로만 시연 |
| R6 | 터틀봇3 펌웨어 호환성 문제 | 2차 진행 지연 | W2 D9에 펌웨어 업데이트 검증, 안 되면 ROS2 Foxy로 다운그레이드 검토 |
| R7 | RealSense 드라이버 충돌 | Vision Node 동작 불가 | librealsense 특정 버전 고정, USB 3.0 포트 확인, 드라이버 먼저 동작 검증 후 진행 |
| R8 | Dobot SDK가 ROS2 Humble 비호환 | 4차 진행 막힘 | 4차는 stretch goal이므로 포기, 정지 픽업까지만 또는 컨셉만 발표 |

### Low Impact / High Likelihood (감수 또는 최소 대응)

| # | 위험 | 영향 | 대응 |
|---|---|---|---|
| R9 | Tailscale 일시 끊김 | 음성 명령 일시 작동 안 함 | 안전 키워드는 LLM 우회, 명령 카탈로그 키워드 fallback (W4) |
| R10 | llama.cpp 응답 지연 | 음성 응답 체감 답답 | 5초 타임아웃 + fallback, 모델 크기 축소 검토 |
| R11 | pyttsx3 한국어 음질 부족 | 시연 인상 약함 | Edge TTS로 교체 (반나절 작업) |
| R12 | 시연 환경 마이크 노이즈 | STT 인식률 저하 | 시연 직전 마이크 확인, push-to-talk 사용, 음량 조정 |

### Low Impact / Low Likelihood (모니터링만)

- 라즈베리파이 5 발열 — 방열판/팬 장착, 시연 길지 않으면 문제 없음
- SQLite 동시 쓰기 — 우리 시스템에선 단일 writer라 무관
- Wi-Fi 다른 장치와 채널 충돌 — 5GHz 사용으로 대체로 회피

## 우회 가능 영역과 우회 불가 영역

### 우회 불가 — 사전 검증으로만 대응

다음은 우회 수단이 없으므로 W1에 반드시 검증한다.

- **컨베이어 키트 제어** — 매뉴얼/회로 추적으로 핀 확정. 실패 시 일정 재조정
- **터틀봇3 와플 동작** — ROS2 Humble 호환 펌웨어 적재 확인
- **RealSense + Ubuntu 22.04** — 표준 조합이지만 USB 3.0 포트 필수
- **집 데스크톱 ↔ Tailscale 연결** — 네트워크 환경 확인

### 우회 가능 — 대안 경로 미리 준비

- LLM: 클라우드 API로 대체 (Anthropic, OpenAI 키 발급)
- 음성 처리: 브라우저 Web Speech API로 STT 대체 가능 (정확도 감소)
- TTS: pyttsx3 → Edge TTS → 브라우저 SpeechSynthesis 순으로 fallback
- 동적 장애물 회피: 단순 자율주행만으로 마감 가능
- 4차 Dobot: stretch goal, 완전 포기 가능

## W1 우선 검증 항목 (D1-D2)

W1 첫 이틀에 다음 모든 항목을 통과해야 본격 개발 시작.

- [ ] Ubuntu 22.04에 ROS2 Humble 설치, talker/listener 예제 동작
- [ ] 컨베이어 키트 매뉴얼 수령 또는 핀 매핑 추적 완료
- [ ] 라즈베리파이 5에 OS 설치, gpiozero로 LED blink 동작
- [ ] 보드 12V 인가 후 LED 점등, 노란 버튼 입력 GPIO에서 감지
- [ ] 서보 1회 회전 확인
- [ ] 스테퍼 1회 회전 확인
- [ ] 노트북에서 RealSense `realsense-viewer` 동작
- [ ] 노트북에서 ROS2의 realsense-ros 노드 실행 → 토픽 publish 확인
- [ ] 터틀봇3 와플 부팅, 와플 라즈베리파이 SSH 접속
- [ ] 터틀봇3 ROS2 Humble 펌웨어 적재 확인
- [ ] 노트북에서 `ros2 topic list`로 터틀봇 토픽 발견
- [ ] 집 데스크톱 llama.cpp 서버 동작, Tailscale로 노트북에서 호출 성공

이 중 하나라도 미통과 시 그 항목을 우선 해결, 다른 작업 미루는 것이 원칙.

## 일정 vs 품질 트레이드오프

만약 W3말 체크포인트 미달 시 다음 우선순위로 마감한다.

1. **반드시 살아있어야 함**: 1차 (컨베이어 + 검출 + 분류) — 시연 영상 메인 컷
2. **가능하면 살림**: 2차 (터틀봇 자율 이동) — 음성 트리거까지
3. **시간 남으면**: 3차 (SLAM 매핑, 자율주행)
4. **포기 가능**: 동적 장애물 회피, 4차 Dobot, 명령 카탈로그 확장

품질 vs 양 선택 시 **양보다 질** 우선. "1차만 완벽하게 시연 가능"이 "1~3차 다 어설프게 동작"보다 나음.
